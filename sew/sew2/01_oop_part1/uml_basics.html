Grundlagen UML

und Klassendiagramme

Was ist UML?

  • UML = Unified Modeling Language
  • Standardisierte Sprache zur Beschreibung von Software
  • Besteht aus verschiedenen Diagrammtypen
  • Wird vor allem für die Planung und Dokumentation genutzt
  • Keine Programmiersprache, sondern ein Modellierungswerkzeug

Wozu dient UML?

  • Hilft, komplexe Programme besser zu verstehen
  • Erleichtert die Kommunikation im Team
  • Dient als „Bauplan“ für den späteren Code
  • Zeigt Strukturen und Zusammenhänge übersichtlich

Arten von UML-Diagrammen

  • Strukturdiagramme: z. B. Klassendiagramm
  • Verhaltensdiagramme: z. B. Aktivitätsdiagramm, Sequenzdiagramm

„Strukturdiagramme zeigen, was da ist — Verhaltensdiagramme zeigen, was passiert.“

Wir schauen uns zum Start das Klassendiagramm an.

Klassendiagramme

  • Zeigen den Aufbau von Klassen
  • Zeigen Attribute (Eigenschaften) und Methoden
  • Zeigen Beziehungen zwischen Klassen
  • Jede Klasse im Diagramm → eine Java-Klasse

Aufbau einer Klasse (Beispiel)

+--------------------+
|   Person           |
+--------------------+
| - name : String    |
| - alter : int      |
+--------------------+
| + sagHallo() : void|
+--------------------+
  • Oberer Block: Klassenname
  • Mittlerer Block: Attribute (mit Typ)
  • Unterer Block: Methoden (mit Rückgabetyp)

Sichtbarkeit in UML

  • + public (öffentlich): Überall sichtbar
  • - private (privat): Nur innerhalb der Klasse sichtbar
  • # protected (geschützt): In der Klasse und in Unterklassen sichtbar
  • ~ package (Paket-Sichtbarkeit): Innerhalb desselben Pakets sichtbar

Beispiel: Sichtbarkeit

+----------------------------------+
|   Konto                          |
+----------------------------------+
| - kontonummer : int              |
| # inhaber: String                |
| ~ bankleitzahl : int             |
+----------------------------------+
| + einzahlen(betrag: float) : void|
+----------------------------------+
  • - kontonummer: privat
  • # inhaber: protected
  • ~ bankleitzahl: package
  • + einzahlen(): public

Beziehungen zwischen Klassen

Klassen stehen selten allein. Sie arbeiten oft zusammen. Diese Zusammenarbeit wird durch Beziehungen dargestellt.

  • Dependency: Eine Klasse "verwendet" eine andere (z.B. als Parameter)
  • Assoziation: Eine allgemeine Beziehung ("kennt ein")
  • Aggregation : Eine spezielle Art der Assoziation, die eine "hat ein"-Beziehung ausdrückt. (Teil des Ganzen, aber unabhängig)
  • Komposition: Eine spezielle Art der Assoziation, die eine starke "besteht aus"-Beziehung ausdrückt. (Teil ist vom Ganzen abhängig)

Abhängigkeit (Dependency)

Eine Klasse verwendet eine andere, wenn sie diese z.B. als Parameter in einer Methode benötigt. Die Beziehung ist meist kurzlebig.

+-------------------+      uses      +-------------------+
|   Drucker         |-------------->|   Dokument        |
+-------------------+               +-------------------+
| + drucken(d: Dok) |               |                   |
+-------------------+               +-------------------+

Verwendung: Java-Beispiel


public class Drucker {
    // Der Drucker verwendet ein Dokument, besitzt es aber nicht.
    // Das Dokument wird nur für die Dauer der Methode "gekannt".
    public void drucken(Dokument dokument) {
        System.out.println("Drucke: " + dokument.getInhalt());
    }
}

public class Dokument {
    private String inhalt;
    // ... Konstruktor, Getter ...
}

Beziehung: Assoziation

Zwei Klassen sind miteinander verbunden, aber keine ist ein Teil der anderen.

+-----------+       +---------+
|  Schüler  |-------| Lehrer  |
+-----------+       +---------+
  • Ein Schüler "kennt" einen Lehrer (z.B. hat Unterricht bei ihm).
  • Ein Lehrer "kennt" einen Schüler (z.B. unterrichtet ihn).
  • Beide Klassen existieren unabhängig voneinander.

Assoziation: Java-Beispiel


// Ein Lehrer kennt seine Schüler und umgekehrt.
public class Lehrer {
    private List<Schueler> schueler;
    // ...
}

public class Schueler {
    private Lehrer lehrer;
    // ...
}

Multiplizität bei Assoziationen

Gibt an, wie viele Objekte einer Klasse mit wie vielen Objekten einer anderen Klasse in Beziehung stehen.

+-----------+ 1..*    1 +-----------+
|  Schüler  |-----------| Lehrer    |
+-----------+           +-----------+
  • 1: Genau ein Objekt.
  • * oder 0..*: Beliebig viele Objekte (auch keines).
  • 1..*: Mindestens ein Objekt.
  • 0..1: Kein oder genau ein Objekt.

Im Beispiel: Ein oder mehrere Schüler lernen bei genau einem Lehrer.

Aggregation ("hat ein")

Eine Klasse ist Teil einer anderen Klasse, kann aber auch ohne sie existieren.

+-----------+<>----
|  Abteilung|     |
+-----------+     |   +-------------+
              1   * - | Mitarbeiter |
                      +-------------+
  • Eine Abteilung "hat ein" oder mehrere Mitarbeiter.
  • Der Mitarbeiter ist Teil der Abteilung.
  • Wird die Abteilung aufgelöst, existiert der Mitarbeiter weiter.
  • Symbol: Leere Raute (an der Seite des "Ganzen").

Aggregation: Java-Beispiel


public class Abteilung {
    // Die Abteilung "hat" Mitarbeiter, aber erstellt sie nicht selbst.
    private List<Mitarbeiter> mitarbeiter;

    public Abteilung(List<Mitarbeiter> mitarbeiter) {
        this.mitarbeiter = mitarbeiter;
    }
}

public class Mitarbeiter {
    // Kann auch ohne Abteilung existieren.
}

Komposition ("besteht aus")

Eine starke Form der Aggregation. Der Teil kann ohne das Ganze nicht existieren.

+-------+<#>----
| Haus  |      |
+-------+      |  +------+
        1    1..* | Raum |
                  +------+
  • Ein Haus "besteht aus" einem oder mehreren Räumen.
  • Der Raum ist ein fester Bestandteil des Hauses.
  • Wird das Haus abgerissen, existiert der Raum auch nicht mehr.
  • Symbol: Ausgefüllte Raute (an der Seite des "Ganzen").

Komposition: Java-Beispiel


public class Haus {
    // Das Haus "besitzt" seine Räume.
    // Die Räume werden vom Haus selbst erstellt.
    private List<Raum> raeume;

    public Haus() {
        this.raeume = new ArrayList<>();
        this.raeume.add(new Raum("Wohnzimmer"));
        this.raeume.add(new Raum("Küche"));
    }
}

public class Raum {
    // Existiert logisch nicht ohne ein Haus.
    private String name;
    public Raum(String name) { this.name = name; }
}

Fazit

  • UML hilft, Software zu planen und zu verstehen
  • Klassendiagramme zeigen Aufbau und Beziehungen von Klassen
  • Sie sind eine Brücke zwischen Idee und Code

🔑 Planung ist der erste Schritt zu gutem Code!
