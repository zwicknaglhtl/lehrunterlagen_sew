Kapselung

mittels Access Modifier und Getter/Setter-Methoden

Access Modifier in Java

  • Steuern die Sichtbarkeit von Klassen, Attributen und Methoden.
  • Wichtig für Kapselung und Datenintegrität.

Die 4 Access Modifier

  • public: Überall zugänglich.
  • protected: Zugänglich innerhalb des Pakets und in Unterklassen.
  • default: Nur innerhalb desselben Pakets zugänglich (kein explizites Schlüsselwort).
  • private: Nur innerhalb der deklarierenden Klasse zugänglich.

Beispiele für Access Modifier

public


public class ClassExample {
    public int publicVar = 10;
    public void publicMethod() { /* ... */ }
}
   

  public class OtherClass {
    
    public void access() {
      // Zugriff von überall möglich  
      ClassExample obj = new ClassExample();
      System.out.println(obj.publicVar); // Zugriff ist erlaubt
      obj.publicMethod(); // Zugriff ist erlaubt
    }
}

Beispiele für Access Modifier

private


public class ClassExample {
    private int privateVar = 40;
    private void privateMethod() { /* ... */ }
    public int getPrivateVar() { return privateVar; }
}
 

  public class OtherClass {
    public void access() {
      ClassExample obj = new ClassExample();
      //obj.privateVar; // Fehler - kein Zugriff von außen möglich!
      System.out.println(obj.getPrivateVar()); //Zugriff über Methode erlaubt
    }
  }

Beispiele für Access Modifier

protected

(Unterklassen lernen wir noch!)


package com.example.package1;
public class ClassExample {
    protected int protectedVar = 20;
   

 //Im selben Paket oder in Unterklassen zugänglich
 package com.example.package2;
 class SubClass extends ClassExample {
     void access() { System.out.println(protectedVar); }
 }

Beispiele für Access Modifier

(default)


package com.example.package1;
public class ClassExample { 
    int defaultVar = 30; // Kein Modifier
}
      

 
 package com.example.package2;
 public class OtherClass {
    public void access() {
      ClassExample obj = new ClassExample();
      //obj.defaultVar; // Fehler - Nur innerhalb von com.example.package1 zugänglich
    }
  }

Access Modifier bei Klassen

  • public class: Die Klasse ist von überall sichtbar und kann von jedem anderen Code verwendet werden.
  • default (kein Modifier): Die Klasse ist nur innerhalb desselben Pakets sichtbar (package-private).
  • private/protected: Nicht für Top-Level-Klassen erlaubt!

  // Datei: MeineKlasse.java
  public class MeineKlasse { }

  // Datei: PaketKlasse.java
  class PaketKlasse { }

Nur innere Klassen (Klassen innerhalb einer anderen Klasse) können private oder protected sein.

Getter- und Setter-Methoden

  • Standardisierte Methoden, um auf private Attribute zuzugreifen oder diese zu ändern.
  • Getter (Accessoren): Liefern den Wert eines Attributs zurück (z.B. getKontostand()).
  • Setter (Mutatoren): Ermöglichen das Setzen eines Attributs, oft mit Validierung (z.B. setKontostand(double neuerKontostand)).
  • Ermöglichen kontrollierten Zugriff und wahren die Kapselung.

Namenskonventionen für Getter & Setter

  • Folgen einem festen Muster in Java.
  • Getter: Beginnen mit get gefolgt vom Attributnamen in CamelCase (z.B. getName(), getAlter()).
    • Für boolean-Attribute kann der Getter auch mit is beginnen (z.B. isAktiv()).
  • Setter: Beginnen mit set gefolgt vom Attributnamen in CamelCase (z.B. setName(String name), setAlter(int alter)).
    • Der Parametername ist oft identisch mit dem Attributnamen (this zur Unterscheidung).

Beispiel


public class Person {
    private String name;
   
    // Getter für name
    public String getName() {
        return name;
    }

    // Setter für name
    public void setName(String name) {
      if (name != null && !name.isEmpty()) {
          this.name = name;
      } else {
          System.out.println("Name darf nicht leer sein!");
      }
    }
}

Beispiel 2


public class Produkt {
    private boolean verfuegbar;

    public boolean isVerfuegbar() { // Getter für boolean
        return verfuegbar;
    }

    public void setVerfuegbar(boolean verfuegbar) { // Setter
        this.verfuegbar = verfuegbar;
    }
}

Was ist eine Schnittstelle?

Als Schnittstelle einer Klasse bezeichnet man alle öffentlich sichtbaren Bestandteile einer Klasse.

Eine Schnittstelle definiert, wie andere Klassen mit einer Klasse interagieren können, ohne die internen Details zu kennen.

Application Programming Interface (API)

  • Eine API ist die gesamte Sammlung von Schnittstellen, die ein größeres System (Modul, Framework, Bibliothek) nach außen bereitstellt.
  • Eine API besteht also aus vielen Schnittstellen von Klassen (oder Interfaces, Modulen, Paketen).
  • Die Java API ist ein Beispiel für eine umfangreiche API.

Kriterien einer guten Schnittstelle (Auszug)

  • ✅ Methodennamen sollten selbsterklärend sein (addItem(), removeUser() statt doStuff()).
    • Ein Nutzer der Klasse sollte nur vom Methodenkopf verstehen können, was die Schnittstelle leistet.
  • ✅Methoden einer Klasse sollten logisch zusammenpassen und ein einheitliches Verantwortungsgebiet abbilden (Single Responsibility Principle).

Kriterien einer guten Schnittstelle (Auszug)

  • ✅Von außen sieht man nicht wie die Klasse etwas macht, sondern nur was sie macht.
    • Die konkrete Implementierung innerhalb der Klasse soll man ohne Auswirkungen auf den restlichen Code ändern können.
  • ✅ Nur das nach außen geben, was wirklich nötig ist → möglichst wenige public-Member.
    • Prinzip der Kapselung - siehe folgende Folien.

Was ist Kapselung?

  • Verbergen der inneren Implementierungsdetails
  • Zugriff nur über definierte Schnittstellen (Methoden)
  • Schützt Daten vor ungewolltem Zugriff von außen.

Kapselung wird erreicht über Access Modifier und setter-/getter Methoden.

Beispiel in Java


public class Konto {
    private double kontostand; 

    public void setKontostand(double kontostand) {
        this.kontostand = kontostand;
    }

    public double getKontostand() {
        return kontostand;
    }
}

→ kontostand ist gekapselt und nur über Methoden zugänglich

Warum ist Kapselung wichtig?

Ohne Kapselung kann jeder Code beliebig Daten verändern:


public class Konto {
    public double kontostand = 0; //soll im Beispiel nicht negativ sein dürfen.
}

Konto k = new Konto();
k.kontostand = -9999; // geht! Aber sollte in unserem Beispiel nie so sein.

❌ Jetzt ist ein negativer Kontostand möglich!

Warum ist Kapselung wichtig?

Mit Kapselung wird der Zugriff kontrolliert:


// Bessere Lösung: mit Kapselung
public class Konto {
    private double kontostand = 0;

       public void setKontostand(double kontostand) {
        if (kontostand >= 0) {
            this.kontostand = kontostand;
        } else {
            System.out.println("Kontostand darf nicht negativ sein!");
        }
    }
}

✅ Jetzt ist kein negativer Kontostand mehr möglich!

Vorteile der Kapselung

  • Erlaubt Überprüfung(Validierung) von Eingabewerten innerhalb der Klasse.
  • Erlaubt Änderungen an der Implementierung, ohne Aufrufe aus anderen Klassen ändern zu müssen.
  • Die Kontrolle über die Logik und die Daten bleibt innerhalb der Klasse.
  • Verbessert die Verständlichkeit, Wartung und Datensicherheit des Codes und reduziert die Abhängigkeiten zwischen Klassen.
